# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A WordPress 7.0.2 site (`edtech`) running in DDEV. WordPress core lives at the repo root (`docroot: ""`), so core files and project files share one directory tree — the `.gitignore` is what separates them.

Environment: PHP 8.5, MariaDB 10.11, nginx-fpm, WP-CLI 2.12. Site URL: `https://edtech.ddev.site`.

## Commands

Everything runs inside the container — always prefix with `ddev`. Never run `php`, `composer`, `wp`, or `mysql` on the host.

```bash
ddev start / ddev stop / ddev restart   # lifecycle
ddev describe                            # URLs, ports, service status
ddev ssh                                 # shell in web container (/var/www/html)
ddev logs -s web -f                      # PHP/nginx errors
ddev launch                              # open site
ddev launch /wp-admin                    # open admin
ddev mailpit                             # captured outbound mail
```

WP-CLI and database:

```bash
ddev wp <command>                        # e.g. ddev wp plugin list, ddev wp user list
ddev mysql                               # DB shell (db/db, root/root)
ddev export-db --file=dump.sql.gz
ddev import-db --file=dump.sql.gz
ddev snapshot                            # DB snapshot before risky migrations
ddev snapshot restore --latest
```

Debugging:

```bash
ddev xdebug on / off / status            # off by default; leaving it on slows every request
```

Note: WP-CLI 2.12 emits PHP 8.5 deprecation warnings on every invocation (from its bundled `react/promise`). They are noise from the phar, not from this codebase — commands still exit 0. To read clean output, pipe through `tail` or suppress: `ddev exec php -d error_reporting='E_ALL & ~E_DEPRECATED' /usr/local/bin/wp-cli <cmd>`.

There is no build, lint, or test tooling configured yet. Do not invent commands for it.

## Configuration model

Config is generated by DDEV and must not be hand-edited where marked:

- `wp-config.php` and `wp-config-ddev.php` both carry a `#ddev-generated` header. DDEV overwrites them on `ddev start` unless that comment line is removed. Put custom constants in `wp-config.php` between the `/* Add any custom values... */` and `/* That's all, stop editing! */` markers only after deciding to take ownership of the file.
- DB credentials, `WP_HOME`, `WP_SITEURL`, `$table_prefix`, and `WP_DEBUG` all come from container env vars via `wp-config-ddev.php` — they are never hardcoded and must stay that way so the project stays portable.
- `WP_DEBUG` is **on** by default in DDEV. Disable with `ddev config --web-environment-add=WP_DEBUG=false`.
- Environment changes (PHP version, database, hostnames) go in `.ddev/config.yaml` or a `.ddev/config.*.yaml` override, followed by `ddev restart`.

## What is and isn't tracked in git

`.gitignore` follows GitHub's WordPress template: core (`/wp-admin/`, `/wp-includes/`, `/wp-*.php`, `/index.php`), `wp-config.php`, `/wp-content/uploads/`, bundled `twenty*` themes, and `hello.php` are all excluded. **Project code is what lives under `wp-content/` — custom themes, plugins, and mu-plugins.** That is where changes belong.

When adding a bundled/third-party plugin that is installed rather than authored here, add it to `.gitignore` rather than committing its tree.

## Conventions

- WordPress coding standards (tabs, `snake_case` functions, Yoda conditions) — match the surrounding core/plugin style.
- Prefix all custom functions, hooks, options, and post types to avoid collisions with core and plugins.
- Escape on output (`esc_html`, `esc_attr`, `esc_url`), sanitize on input, use `$wpdb->prepare()` for any direct SQL, and verify nonces + capabilities on every form and AJAX/REST handler.
- Never edit core files at the repo root or in `wp-admin/`/`wp-includes/` — they are gitignored and destroyed by every WordPress update.
